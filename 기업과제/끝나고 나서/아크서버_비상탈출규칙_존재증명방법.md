# 아크(Ark) 비상 탈출 규칙의 존재 증명 방법

## 왜 이걸 알아야 하는가

아크(Ark) 서버가 먹통이 되어도 내 돈을 꺼낼 수 있다는 건 알겠는데,  
**그게 실제로 어떻게 가능한 건지**가 궁금할 수 있다.

누군가 블록체인에 "비상 탈출 가능합니다"라고 써놓고  
나중에 "이거 내 거 맞아요!" 하고 우기는 방식이 아닐텐데  
그럼 실제로는 어떻게 동작하는 걸까?

한 문장으로 요약하면:

> **비상 탈출 규칙은 처음부터 금고 주소 안에 수학적으로 박혀 있어서,  
> 나중에 꺼낼 때 그 수학적 증거를 그대로 제시하면 된다.**

아래에서 그 과정을 하나씩 풀어본다.

---

## 1단계 — 규칙은 어떤 언어로 쓰이는가

"비상 탈출 가능합니다"라는 자연어 문장을 비트코인 네트워크는 읽을 수 없다.  
비트코인은 자신만의 명령어 체계를 갖고 있고,  
규칙은 이 언어로 작성된다.

비상 탈출 규칙을 대략 번역하면 이런 모양이다:

```
<사용자 공개키>   OP_CHECKSIG              → "이 사람 서명이 맞는가?"
<특정 시간 조건>  OP_CHECKSEQUENCEVERIFY   → "지정된 시간(또는 블록 수)이 지났는가?"
OP_DROP OP_TRUE                            → "두 조건 다 맞으면 통과"
```

이 명령어 묶음 하나가 곧 하나의 **지출 조건(Spend condition)** 이 된다.  
아크 금고 주소에는 이런 조건이 최소 2개 들어간다.

- **경로 A** — 서버와 협력해서 자금을 움직이는 정상 경로
- **경로 B** — 서버 없이 혼자 탈출하는 비상 경로

---

## 2단계 — 두 규칙을 어떻게 하나의 주소에 숨기는가

두 규칙을 그대로 블록체인에 올리면 내용이 전부 공개된다.  
아크는 이것을 **해시 함수(Hash function)** 로 압축한 다음, **머클 트리(Merkle Tree)** 구조로 하나로 묶는다.

해시 함수는 어떤 내용을 넣어도 고정된 길이의 암호 문자열로 바꿔주는 함수인데,  
중요한 특징은 **결과물만 보고 원본을 역추적할 수 없다**는 것이다.

```
경로 A 스크립트  →  (SHA-256)  →  Hash_A
                                           \
                                            (SHA-256)  →  Hash_AB  ← 머클 루트
경로 B 스크립트  →  (SHA-256)  →  Hash_B  /
```

이렇게 만들어진 **최종 해시값(= 머클 루트, Merkle Root)** 을  
금고 주소를 만들 때 공개키에 수학적으로 더해버린다(이 과정을 **Tweak** 라고 한다).

결과물인 탭루트(Taproot) 주소는 겉보기엔 평범한 비트코인 주소처럼 보이지만,  
안에는 두 경로의 규칙이 해시 형태로 조용히 박혀 있다.  
어떤 규칙이 숨어 있는지는 **실제로 사용하기 전까지는 아무도 모른다.**

---

## 3단계 — 탈출할 때 무엇을 제출하는가

비상 탈출 트랜잭션을 메인넷에 보낼 때, 단순히 "나 탈출할게요"만 외치는 게 아니라  
아래 **3가지 데이터** 를 함께 제출한다.

| 제출 항목 | 역할 |
|---|---|
| **① 경로 B 스크립트 원본** | "이게 내가 주장하는 비상 탈출 규칙입니다" |
| **② 내 디지털 서명** | "그 규칙의 조건을 내가 만족시킵니다" |
| **③ 컨트롤 블록 (Control Block)** | "이 규칙이 처음부터 이 주소 안에 있었다는 수학적 증거입니다" |

③번 컨트롤 블록이 핵심이다.  
여기에는 짝꿍인 **Hash_A** (경로 A의 해시값)가 들어 있다.  
"나는 경로 B(규칙 써있음)를 공개하고, 짝꿍 Hash_A도 내놓을 테니 네가 직접 계산해봐" 하고 내미는 것이다.

---

## 4단계 — 비트코인 노드가 검증하는 과정

채굴자(노드)가 이 데이터를 받으면 다음 순서로 계산한다.

```
① 경로 B 스크립트 원본을 SHA-256으로 해시   →  Hash_B 계산
② Hash_B + 컨트롤 블록의 Hash_A 를 합쳐서 다시 SHA-256  →  Hash_AB 계산
③ 이 Hash_AB 를 원래 금고 주소의 공개키에서 역산
④ 기존 주소와 일치하면 → ✅  일치하지 않으면 → ❌
```

일치하면 노드는 이렇게 판단한다:

> **"이 경로 B 스크립트는 이 금고 주소가 만들어질 때부터 안에 들어있던 게 맞네.  
> 수학적으로 확인됨 땅땅."**

그다음 스크립트 안의 명령어들을 실제로 실행해서  
서명이 유효한지, 시간 조건이 충족됐는지 최종 확인하고 자금을 내어준다.

---

## 정리

비상 탈출의 핵심은 **"나중에 주장"하는 게 아니라, "처음부터 증명해놓는" 구조**다.

금고 주소를 만드는 순간, 비상 탈출 규칙은 이미 그 주소의 수학적 구성 요소로 박혀 있다.  
탈출할 때는 그 규칙 원본과 수학적 증거(컨트롤 블록)를 꺼내 보이기만 하면 되고,  
비트코인 노드가 직접 계산해서 진짜인지 확인한다.

## 📌 추가 학습이 필요한 부분

이 문서에서는 개념 흐름을 이해하는 것에 집중했기 때문에,  
아래 항목들은 키워드만 등장하고 원리가 생략되어 있다.  
더 깊이 이해하고 싶다면 아래를 참고하자.

### 1. 비트코인 스크립트의 스택 실행 방식
Op-code들은 실제로 **스택(Stack) 자료구조** 위에서 순서대로 쌓이고 꺼내지며 실행된다.  
`OP_CHECKSIG`, `OP_CHECKSEQUENCEVERIFY` 같은 명령어가 구체적으로 어떻게 작동하는지는 생략했다.  
→ 참고: [Bitcoin Script — Bitcoin Wiki](https://en.bitcoin.it/wiki/Script)

### 2. 탭루트(Taproot)의 Tweak 연산
공개키에 머클 루트를 "더한다"고 표현했는데, 실제로는 **타원 곡선(Elliptic Curve)** 위에서 일어나는 연산이다.  
이것이 왜 안전하고 역추적이 불가능한지는 타원 곡선 암호학(ECC) 지식이 필요하다.  
→ 참고: [BIP-341 (Taproot)](https://github.com/bitcoin/bips/blob/master/bip-0341.mediawiki)

### 3. SHA-256 해시 함수의 보안 근거
"역추적 불가"라고만 설명했는데, 정확히는 **역상 저항성(preimage resistance)** 과  
**충돌 저항성(collision resistance)** 이라는 수학적 성질이 이를 보장한다.  
→ 참고: [SHA-2 — Wikipedia](https://en.wikipedia.org/wiki/SHA-2)

### 4. 경로가 3개 이상일 때의 머클 트리
이 문서는 경로 A/B 두 개짜리 단순 구조를 예시로 썼다.  
실제로 여러 지출 조건이 있을 때 트리가 깊어지고 Merkle Proof 길이가 늘어나는 방식은 생략했다.  
→ 참고: [Merkle Tree — Wikipedia](https://en.wikipedia.org/wiki/Merkle_tree)

### 5. 컨트롤 블록(Control Block)의 실제 데이터 구조
이 문서에서는 "Hash_A를 함께 낸다"고 단순화했지만,  
실제 Control Block은 **버전 바이트 + 내부 공개키(internal key) + Merkle Proof 경로 전체** 를 포함한다.  
→ 참고: [BIP-341 §Script validation rules](https://github.com/bitcoin/bips/blob/master/bip-0341.mediawiki#script-validation-rules)
